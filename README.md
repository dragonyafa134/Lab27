## Главные выводы

1. **Минимализм и производительность**: Minimal API в ASP.NET Core позволяет создавать веб-серверы с минимальным количеством кода, при этом сохраняя высокую производительность за счёт упрощённого конвейера обработки запросов по сравнению с полноценным MVC.

2. **Middleware — мощный инструмент обработки**: Порядок регистрации middleware критически важен — они выполняются в цепочке в том порядке, в котором были добавлены через `app.Use()`. Это позволяет реализовывать сквозную функциональность (логирование, аутентификацию, сжатие ответов) без изменения кода маршрутов.

3. **Автоматическая сериализация JSON**: ASP.NET Core автоматически преобразует объекты C# в JSON и использует camelCase для имён свойств (PascalCase → camelCase), что соответствует стандартам JavaScript/TypeScript и улучшает совместимость с фронтенд-фреймворками.

## Вопросы для самопроверки

### 1. Чем Minimal API отличается от MVC в ASP.NET Core?

| Характеристика | Minimal API | MVC |
|----------------|-------------|-----|
| **Объём кода** | Минимальный, всё в одном файле (`Program.cs`) | Разделён на контроллеры, модели, представления |
| **Подход** | Функциональный (лямбда-обработчики) | Объектно-ориентированный (контроллеры с методами) |
| **Сложность** | Низкий порог входа, идеален для микросервисов и малых API | Высокий, подходит для крупных enterprise-приложений |
| **Производительность** | Выше за счёт меньшего количества промежуточных слоёв | Ниже из-за дополнительных этапов (фильтры, model binding) |
| **Функциональность** | Ограниченная (базовая валидация, DI) | Расширенная (фильтры, middleware, helpers, tag helpers) |

**Вывод**: Minimal API выбирают для простоты и скорости, MVC — для структурированности и масштабируемости.

### 2. Что произойдёт, если поставить `app.MapGet(...)` до `app.Use(...)`?

Middleware, зарегистрированный **после** маршрутов, **не будет выполнен** для этих маршрутов. Конвейер обработки выглядит так:


```
Запрос → [Middleware 1] → [Middleware 2] → [Маршрут] → Ответ
```


Если `MapGet` находится **до** `Use`, то запрос сразу попадает в обработчик маршрута, минуя middleware, объявленные позже. Правильный порядок:

```csharp
app.Use(async (context, next) => { /* логирование */ await next(); });
app.MapGet("/", () => "Hello");  // Middleware сработает
```

3. Почему ASP.NET Core преобразует PascalCase в camelCase при сериализации JSON?
Это сделано для соответствия стандартам JavaScript/TypeScript и сложившимся конвенциям веб-API:

В JavaScript принято использовать camelCase для имён свойств объектов (firstName, userAge), а не PascalCase (FirstName, UserAge), как в C#.

Большинство фронтенд-фреймворков (React, Angular, Vue) и библиотек ожидают именно camelCase.

Формат JSON не имеет строгого стандарта именования, но camelCase стал де-факто индустриальным соглашением.

При необходимости это поведение можно изменить, установив PropertyNamingPolicy = null в настройках JSON-сериализации.
